# Insurance KPI Ontology Mapping Tool - Full LLM Version

This package contains a FastAPI backend and React/Vite frontend for mapping report KPIs to the Life & Annuity actuarial reserving ontology.

This version is **completely LLM/API-driven** for mapping logic:

- LLM selects the top-N ontology candidates for every report KPI.
- LLM assigns similarity scores and actuarial rationales.
- LLM performs Step 4 formula/definition validation.
- LLM performs Step 5 pairwise formula comparison.
- LLM creates missing-KPI recommendations.
- The backend does **not** copy the reference/golden workbook.
- The backend does **not** silently fall back to deterministic scoring if the API fails.

Python still performs structural work only: loading files, extracting report KPI rows, reading ontology rows, creating pairwise combinations after LLM best mappings, report-wise gap calculations, and Excel export.

## Backend setup

```powershell
cd insurance_kpi_ontology_tool/backend
python -m venv .venv
Set-ExecutionPolicy -Scope Process -ExecutionPolicy Bypass
.venv\Scripts\Activate.ps1
python -m pip install --upgrade pip setuptools wheel
pip install -r requirements.txt
copy .env.example .env
```

Edit `backend/.env`.

### OpenAI

```env
LLM_PROVIDER=openai
OPENAI_API_KEY=your_key_here
OPENAI_BASE_URL=https://api.openai.com/v1
OPENAI_MODEL=gpt-4o-mini
```

### Azure OpenAI

```env
LLM_PROVIDER=azure_openai
AZURE_OPENAI_API_KEY=your_key_here
AZURE_OPENAI_ENDPOINT=https://your-resource-name.openai.azure.com/
AZURE_OPENAI_API_VERSION=2024-10-21
AZURE_OPENAI_DEPLOYMENT=your_deployment_name_here
```

For Azure, `AZURE_OPENAI_DEPLOYMENT` must be the deployment name created in Azure AI Foundry / Azure OpenAI Studio.

Run backend:

```powershell
uvicorn main:app --reload --port 8000
```

Health check:

```text
http://localhost:8000/api/health
```

## Running the tool

Upload:

1. `report_a.json`
2. `report_b.json`
3. `life_annuity_actuarial_reserving_kpi_ontology.xlsx`

Then click **Run Mapping**. Full LLM mode is always on. If the API key, Azure deployment, endpoint, quota, or model is wrong, the backend stops and returns a clear error instead of generating fallback scores.

## Output

The backend generates:

```text
final_output_with_all_steps.xlsx
```

with the same 16-sheet workbook structure:

1. `1. Summary`
2. `2. All Comparisons`
3. `3. Best Mapping`
4. `4. Common Ontology`
5. `5. Step4 Definition Validation`
6. `6. Step5 Pairwise Check`
7. `LLM Pairwise Formula Check`
8. `Best Mapping - EB Details`
9. `Best Mapping - Worksite`
10. `RW Missing KPI Summary`
11. `EB Mapped Ontology KPIs`
12. `EB Missing Ontology KPIs`
13. `EB LLM Missing KPI Recs`
14. `Worksite Mapped Ontology KPIs`
15. `Worksite Missing Ontology KPIs`
16. `Worksite LLM Missing KPI Recs`

Because mappings are now generated by the model, exact scores/rationales can vary by provider/model/temperature. Keep `LLM_TEMPERATURE=0.1` for more stable results.

## UI Note

This package keeps the uploaded enterprise UI exactly as the frontend shell while using the full LLM/API backend. The frontend does not expose a manual LLM toggle because the backend always runs in LLM mode.

Required pipeline logic:

Step 1: Load inputs

* Accept exactly two or more report JSON files and one ontology Excel file from the UI.
* For the current sample, process `report_a.json`, `report_b.json`, and `life_annuity_actuarial_reserving_kpi_ontology.xlsx`.
* Do not rely on hardcoded local Windows paths.
* Do not rely on bundled sample fallback unless explicitly selected in UI.

Step 2: Extract ontology KPIs

* Read sheet `LA_Actuarial_Reserving`.
* Detect header row flexibly.
* Extract:

  * `KPI / Metric Name` as `Ontology KPI Name`
  * `Formula` as `Ontology Formula`
  * `Definition` as `Ontology Definition`
  * Also retain category, synonyms, purpose, source, and confidence internally for dynamic scoring context.
* Expected ontology rows for sample: 98.

Step 3: Extract report KPIs from JSON

* Traverse workbook → sheets → tables → columns.
* Use only formula-based report KPI columns from relevant business summary tables.
* Relevant tables include statutory reserve, tax reserve, GAAP reserve, face amount, cash value, count, exhibit, summary, reserve.
* Exclude raw identifiers and dimensions such as accounting date, product subtype, policy number, triton plancode, ledger product, etc.
* For each KPI, preserve:

  * report file name
  * sheet name
  * table name
  * section title
  * column name
  * formula pattern / formula
  * column definition
  * formula lineage
  * computation type
  * scalar
  * aggregation function
  * sum column
  * inputs
  * group-by fields
  * static filters
  * direct inputs
  * ultimate raw sources
  * fingerprint/context
* Critical: do not collapse rows with the same `Report Col Name` if formulas or sections differ.
* Expected sample extraction:

  * EB Details: 23 KPIs
  * Worksite: 14 KPIs

Step 4: Generate candidate universe

* Cross join every extracted report KPI with every ontology KPI.
* Expected sample universe: 3,626 rows.
* Keep internal hidden fields for report context, table name, section title, sheet name, and formula lineage.

Step 5: Python pre-score and top-5 shortlist

* Implement deterministic Python scoring to select top 5 ontology candidates per unique report KPI.
* Group by composite key:
  `Report A + Report Col Name + Report Formula + table/section/context`
* Do not group by only `Report Col Name`.
* Score using:

  * formula token overlap
  * KPI name overlap
  * ontology synonym overlap
  * statutory / GAAP / tax basis
  * gross / net basis
  * general account / separate account
  * ceded / reinsurance / third-party / captive / recapture / YRT
  * reserve / face amount / cash value / count / premium / claim family
  * aggregation operation: SUM, COUNT, ratio, average
  * sign convention
  * group-by and accounting date filters
* Expected top-5 sample rows: 185.

Step 6: LLM final actuarial scoring

* Send only top-5 rows to the LLM.
* LLM must act as an expert Life & Annuity actuarial reserving specialist.
* Primary comparison basis:

  * report formula
  * report formula business meaning
  * ontology KPI name
  * ontology formula
  * ontology definition
  * formula lineage/context
* Formula logic and actuarial business meaning are more important than name similarity.
* LLM must return structured fields:

  * `similarity_score`
  * `similarity_band`
  * `formula_equivalence_type`
  * `actuarial_rationale`
  * `key_formula_matches`
  * `key_formula_differences`
  * `business_logic_assessment`
  * `confidence_level`
  * `needs_human_review`
* If the LLM fails, preserve the row count and mark the failed rows clearly, but do not drop rows.

Step 7: Best mapping

* Select the highest `similarity_score` ontology row per unique report KPI.
* Use composite identity, not only report column name.
* Expected sample best mapping rows: 37.
* Also split into:

  * `Best Mapping - EB Details`: 23 rows
  * `Best Mapping - Worksite`: 14 rows

Step 8: Common ontology

* From best mapping, identify ontology KPIs that appear in both reports.
* Keep all rows mapped to those common ontology KPIs.
* Expected sample:

  * 10 common ontology KPIs
  * 30 data rows
* Preserve original report/order from best mapping.

Step 9: Step 4 definition validation

* Run LLM validation only for common ontology rows.
* Compare report formula against ontology formula and ontology definition.
* Return:

  * business interpretation of report formula
  * formula similarity score
  * definition similarity score
  * overall step 4 similarity score
  * verdict
  * rationale
* Expected sample: 30 rows.

Step 10: Step 5 pairwise formula comparison

* For each common ontology KPI, compare every unique pair of mapped report formulas.
* Compare formulas mapped to the same ontology KPI, including formulas from the same report and across reports.
* Do not assume common ontology means formulas are identical.
* Expected sample pairwise rows: 35.

Step 11: LLM pairwise formula check

* Run LLM on every Step 5 pair.
* Evaluate whether the two report formulas calculate the same actuarial business metric.
* Return exact structured fields in the golden workbook.

Step 12: Report-wise missing KPI gap analysis

* For each report separately:

  * get ontology KPIs present in that report’s best mapping
  * compare against all 98 ontology KPIs
  * mark missing KPIs for that report only
* A KPI present in EB can still be missing in Worksite.
* Expected sample:

  * EB mapped: 14
  * EB missing: 84
  * Worksite mapped: 11
  * Worksite missing: 87
* Run LLM recommendations for missing KPIs and return priority/actionability fields.
